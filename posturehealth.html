PostureHealth

Gained hands-on experience with an early start-up and learned about the multiple facets of business in the productivity sector.

Overview

PostureHealth is an application that provides real-time feedback for posture and movement. After their first beta launch, their goal was to redesign the in-app experience according to user feedback. They wanted to implement a more intuitive approach, giving their users a more personalized posture coaching experience.

A Continuous Approach

Over the past half year, we ran multiple small sprint-like iterations, starting with a large refactoring iteration in our first sprint. Here’s how that looked on a timeline:

1

Onboarding and Main Features

Redesigned the apps main feature: the posture coaching session and then determined how to walk users through the app.
2

Main Features Additions

Redesigning the reward system following the posture coaching session to increase user incentive.
3

Secondary Features

Reorganizing the way users add and adjust their settings to walk and exercise breaks in their schedule.
4

Settings

Implementing the new structures of the main and secondary features into the settings.
01

Onboarding and Main Features

After making an account, the user is guided through an onboarding process to personalize their experience. The process includes preliminary questions that help understand the user better, followed by an interactive demo that teaches the user how to use the posture coaching feature.

User Flow

Onboarding User Flow (not including Demo flow, see below)

Pulling from user interviews after the app's first launch, we were able to categorize user behavior and create personas. Based off those profiles, we determined four different routes a user could potentially choose, base off of the reason why they are most interested in using the app. The steps of the onboarding process would be presented in an order tailored to the needs the user choose.

We then determined all the features we wanted to present to the user and how that order would be affected by the users choices as they move through the onboarding. Through this flow, we not only wanted to make sure we got all the necessary information to personalize their in-app experience, but also show how our app is doing so by revealing the main points of each feature that add value to our users' lives.

Demo User Flows

While brainstorming the order of how the demo would be presented, we were able to decease the number of steps from seven to five by only including the steps that were most intuitive to the goal of this demo. The introduction of the settings button would not be necessary to this demo since it could be introduced previously in the onboarding process.

Contribution: I created all flow diagrams and high-fidelity user flow (image below). This would allow our design for the demo to be thorough, identifies errors at the start, and save us time in our prototyping phase. I led the design sessions in which we determined the initial layout.

High-Fidelity Demo User Flow

Design

Start Session Screen:
When the user chooses to start their session on the dashboard, this screen will pop up to prompt them to choose their ideal posture position. They select whether they are sitting or standing. Then they "Sync" their movement with the avatar, Zen.

Contribution: I created the switch toggle for choosing the sitting or standing position. I also named the "Sync" button and created an icon with the profile of a person inside a target to emphasize that this button would calibrate their movement with the avatar.


In Session Screen:
After syncing with their avatar, the start session screen shrinks into this smaller screen to begin their posture coaching session.

Contribution: I added the dropdown button to condense the number of buttons on display. This way the user to focus on their coaching session without the excess of buttons.

Onboarding Screens
In addition to the session screens, a cleaner design was implemented to the onboarding process. An animation would accompany each step of the process encourage  the user along.

This is an example of one of those screens (right). This is when the user first chooses which reason they are most interested in using the app (the user personas in the first user flow diagram).

Reflections

The previous start session screen contained paragraphs of text explaining the entire posture coaching session to the user. With the addition of a demo to the end of the onboarding process, we were able to reduce these screens to a much smaller size, saving the user space on their desktop. We wanted the user to practice improving their posture while they were at work and did not want our screens to obstruct their productivity.

02

Main Feature Additions

After users have completed their posture session, the application provides them with a certificate page that illustrates their session statistics.

The Certificate

The original certificate did not appear to motivate users to be consistent with returning to the app and improving their posture.

Orange Theory (far left), iConsole+ (two on far right)

Research

I looked into other fitness tracking apps to see how they presented their workout summaries. I focused on what made these products successful in both statistical and visual presentation.
Orange Theory (left) uses boxed frames with icons and their corresponding statistics underneath them to showcase their users' statistics.
iConsole+ (right) uses a more traditional receipt approach, listing the statistics simply using text.

Design

I redesigned the certificate (right) by incorporating a gold card, to act as a reward to the user for completing their posture session. Their session statistics are now presented in a more visual way, using graphs and icons.

The avatar on the right represents the user and their top posture tendency. Now instead of two static images of the avatar, Zen will animate to show the direction that the user tends to slouch.

I also added rating buttons at the bottom right, next to the other menu buttons to allow the user to provide us with feedback on their app experience.

03

Secondary Features

In addition to the posture coaching session, the app has two types of breaks for the user: walk and exercise. These breaks prompt the user to take a walk or practice an exercise that would help them realign their posture. Users will determine their settings for these breaks during the initial onboarding process.

User Flow

We ultimately wanted to give users the option to choose to implement these breaks or not. We mapped out the different choices they would make when this part of the onboarding process is presented to them (below).

Example of the break questions presented to the user during onboarding.

Design

Users are allowed to determine how often they wanted to be notified to take a break. They could pick between two different scheduling options. If they wanted notifications to appear after a set amount of time, for instance, every 30 minutes, they would select Interval Breaks. If they wanted to choose the exact times they wanted these notification, they would choose Custom Breaks.

Contribution: I designed the questions and the order of them that would be asked of the user during the onboarding process to determine their schedule. I also designed the visualization of the user's break and posture session schedule. I created a calendar view, color coded to indicate whether the session was a break or a posture coaching.

04

Settings

Lastly, the application's settings needed to be reconfigured to accommodate for the changes that were made to it's functions and features.

Research

The original settings (right) was limited and now with the addition of many features, I focused on how other products organized their settings pages to incorporate their many app features as well as include a visual element.

Both Zoom (below left) and Printix (below right) use a tab system to organize their settings into sections, with corresponding icons. This way users are able to find the exact settings they need by looking at the tab categories. Zoom features a video section that contains a video screen where the user can see their setting changes in real time.

Design

The new settings presented a more visual interpretation, using a tab system for organization.

Session Section (left):
I placed each setting into a card with a corresponding animation that updates with each setting change. This gives the user an understanding of which part of their experience that setting controlled.

Schedule Section (right):
This is a direct implementation of the the previous section (Breaks) with the inclusion of the visual calendar, which updates with each setting change.

Activation Section (left):
Here the users can specify which general parts of their body they want to target and improve their posture. I designed a silhouette of the muscle view of the human body (front and back) to let the users choose those targeted parts. I added the circle selectors with a corresponding legend because users may not know the exact muscle groups that is causing their pain points, here we take the guesswork out for them.

Takeaways

This was one of my first experiences working on a real application that involved real user testers. Seeing the feedback and how they reacted to my design decisions taught me the impact of research and applying that feedback. With direct collaboration with the main software developer and co-founder of PostureHealth, I gained insight on how design aligns with business goals and impacts development/implementation.

I enjoyed this experience not only because this was an opportunity to work on a real professional product, but also to respect how the company strives for cross-functional collaboration. I improved my communication and accountability skills through this experience immensely.
